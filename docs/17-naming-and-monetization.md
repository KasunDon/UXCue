# Naming And Monetization Notes

Research date: 2026-07-04

## Recommendation

Recommended product name:

> UXCue

Recommended domains to buy first:

- `uxcue.com`
- `uxcue.dev`
- `uxcue.app`

Optional defensive domain:

- `uxcue.ai`

Why:

- Short and memorable.
- Directly tied to UX/design review without being locked to "bugs" only.
- "Cue" suggests a visual prompt, signal, hint, or next action.
- Works for the product's evolution: local capture, cloud review, GitHub export, MCP/agent bridge.
- Better than UXLens because it sounds less generic and less like an analytics/heatmap product.
- Better than BugCue because it does not limit the product to bugs.
- Better than DefectBrief because it is less negative and less enterprise-dry.

Suggested tagline:

> Capture UX cues. Turn them into agent-ready issues.

Alternate tagline:

> Point at broken UI, create an issue your coding agent can fix.

## Domain Check Method

Domain checks used RDAP lookups where possible.

Interpretation:

- `200`: registered/taken according to RDAP.
- `404`: no RDAP registration record found, likely available.
- `429` or `ERR`: inconclusive.

Important:

- RDAP is a strong signal, not a purchase guarantee.
- Registrar checkout can still show premium pricing, reserved names, temporary locks, or recent purchase changes.
- Buy desired domains quickly if the name is chosen.

## Name Evaluation

### UXCue

Pros:

- Best domain availability.
- Short.
- Friendly.
- Works for UX issues, visual defects, accessibility, copy, and design polish.
- "Cue" implies both reviewer note and agent prompt.
- Does not sound like a generic bug tracker.

Cons:

- "UX" can sound designer-specific and may undersell QA/bug use.
- Pronunciation must be taught once: "U-X Cue."

Verdict:

- Best choice.

### QACue

Pros:

- Good for testing/QA.
- Short and available.
- Broader than UX.

Cons:

- Less emotional/design-oriented.
- "QA" may feel like enterprise testing rather than design helper.

Verdict:

- Good fallback if UXCue feels too design-specific.

### BugCue

Pros:

- Memorable.
- Clear for bug capture.
- Good domain availability.

Cons:

- Too narrow for UX polish, design suggestions, accessibility, and copy.
- Similar mental category to bug reporting tools.

Verdict:

- Strong but not ideal.

### DefectBrief

Pros:

- Very clear agent-work-order meaning.
- Works with markdown/export concept.

Cons:

- Long.
- Negative.
- Sounds more like enterprise QA documentation than a delightful browser helper.

Verdict:

- Good feature name, not best product name.

### UIFlag

Pros:

- Clear.
- Short.
- Available across checked TLDs.

Cons:

- Feels like a simple flag/annotation tool.
- Less strong for agent-ready issue generation.

Verdict:

- Good fallback, not as ownable as UXCue.

## Product Naming System

If product is UXCue:

- Local issue: Cue.
- Review session: Cue Review.
- Export bundle: Cue Pack.
- Cloud: UXCue Cloud.
- MCP: UXCue MCP.
- GitHub sync: UXCue for GitHub.

Use carefully:

- Do not overdo "cue" wordplay.
- Keep issue IDs as `UX-001`, not `CUE-001`, unless rebranding needs it.

## Free Now, Monetize Later

Current business decision:

> UXCue should be free initially, with monetization later through cloud, collaboration, integrations, and agent workflow depth.

This suggests:

- Local extension should stay free.
- Markdown/JSON export should stay free.
- Cloud sync can start free in alpha with quotas.
- Paid tiers should charge for cloud value, team value, storage value, and automation value.

## Proposed Packaging

### Free Local

Price:

- Free forever.

Includes:

- Local projects and review sessions.
- Element capture.
- Screenshots stored locally.
- Metadata capture.
- Issue tracking.
- Markdown/JSON/zip export.
- Manual copy to agents.

Why keep it free:

- Drives adoption.
- Builds trust.
- Differentiates from SaaS-first competitors.
- Creates the top of funnel for cloud.

### Free Cloud Alpha

Price:

- Free during alpha/beta.

Includes:

- Google SSO.
- Limited cloud sync.
- Basic cloud console.
- Screenshot storage with quotas.
- Manual GitHub issue creation if enabled.

Suggested limits:

- 3 cloud projects.
- 10 active sessions.
- 250 cloud issues.
- 1 GB screenshot storage or lower if needed.
- Archived screenshot retention limit.

Goal:

- Validate whether users care about cloud sync/share.

### Solo Pro

Future paid tier.

Includes:

- Higher project/session/issue/storage limits.
- Longer screenshot retention.
- GitHub integration.
- Share links.
- Local MCP server support.
- Advanced export formats.
- Basic a11y/contrast/design checks.

Potential price:

- Low monthly solo dev price.
- Keep flexible until usage and competitor pricing are reviewed.

### Team

Future paid tier.

Includes:

- Shared projects.
- Team members.
- Comments/activity.
- Role permissions.
- GitHub App installation.
- Cloud MCP.
- Audit trail.
- Higher quotas.

Potential buyer:

- Small agencies.
- AI-first product teams.
- Frontend teams doing design QA.

## Monetization Principles

- Do not charge for the core local capture habit.
- Do not force cloud for export.
- Charge for convenience, collaboration, storage, and automation.
- Keep GitHub optional.
- Keep privacy story simple.
- Avoid enterprise sales motion until solo/team usage is proven.
